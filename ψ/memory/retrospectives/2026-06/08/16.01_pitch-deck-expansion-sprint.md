## Honest Feedback (3 Friction Points)

**1. Thai text encoding + Edit tool is a recurrent tax**
Every session that has a context break and Thai-heavy HTML pays this tax: Edit fails, requires Read-with-offset, re-edit. It's not catastrophic but it adds ~5 minutes per file and creates confusion. The fix is known. I need to make it habitual: whenever a context break has occurred, always Read-with-offset before any Edit on Thai-containing HTML. I keep learning this lesson and forgetting to apply it proactively.

**2. The B&E slide should have had a ⚠️ state, not a ✅ state**
When the data for a slide is unverified, the right move is to mark it with a visual "NEEDS SOURCE" tag or placeholder, not render it as if it's confirmed. This session built a polished DUA slide on top of a number that was explicitly flagged as unverified the previous day. The risk is presenting a slide that looks credible but isn't backed by data. If this is for Grand Opening prep, Beejeni needs to know which numbers are confirmed vs. targets.

**3. KOL recommendations are ephemeral**
30 KOL names with channel info were delivered as chat output. No file, no slide, no structured format. This works for quick review but fails as reference material. If Beejeni needs to share the KOL list with a team member or revisit it, it requires scrolling back through a long conversation. Future pattern: always Write a KOL list to `preview/2026-06-02/uficon-kol-recommendations.html` or at minimum a `.md` file.

## Lessons Learned

1. **Read-with-offset before Edit on Thai HTML after context break** — Prevents "String not found" every time.
2. **Unverified numbers need visual markers in slides** — ⚠️ badge or "Target (TBC)" label, not rendered as confirmed.
3. **KOL / list deliverables need a file** — Chat output alone is not persistent enough for reference material.
4. **Segment models need reconciliation before pitch** — Two competing segmentation models in the same deck confuse the story.
